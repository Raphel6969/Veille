# Release 0.2.0 — Phases 1–5 (2026-07-14)

A meaningful vertical slice of the AI Runtime Supervisor: the runtime can now
**observe**, **enforce**, **plan**, **contextualize**, **route**, **optimize**, and
**govern memory** for agent runs — all behind safe opt-in flags by default.

Tag: `v0.2.0`. Event schema version: `0.2.0`.

## What's in this release

| Phase | Capability | Opt-in flag | Status |
|---|---|---|---|
| 1 | Observe + explain (SDK, LangGraph adapter, run summary, OTel, run-explorer) | always on | implemented |
| 2 | Deterministic protection (block / retry / pause / stop + audit) | `SUPERVISOR_ENFORCE` | implemented |
| 3 | Advisory planning (tier, per-step context, capability+tier routing) | `SUPERVISOR_PLAN` | implemented |
| 4 | Adaptive optimization (semantic dedup + idempotent caching) | `SUPERVISOR_OPTIMIZE` (`dry_run` default, `active` to serve) | implemented |
| 5 | Memory lifecycle & retrieval governance (store + scoring + governor, audited expiry) | `SUPERVISOR_MEMORY` | implemented (memory slice) |

All features are **off by default**. Enabling any flag never silently changes the
business outcome of a run unless explicitly activated (`SUPERVISOR_ENFORCE=true`
or `SUPERVISOR_OPTIMIZE_MODE=active`).

## Known limits (read before adopting)

**Scope boundaries (intentional, by design):**

- **No automatic memory deletion.** Expiry only *surfaces* candidates
  (`memory.expired`); `forget_memory` removes explicitly. Autonomous cleanup is a
  later, separately-approved capability.
- **In-run memory only.** `InMemoryMemoryStore` is per-process and non-durable;
  cross-run persistence (Mem0/Letta/Redis/customer RAG) is a backend behind the
  `MemoryBackend` port, not in this release.
- **Metadata-driven memory scoring.** Governance uses recency/usage/provenance/
  confidence (no embeddings). Semantic similarity memory is a future port.
- **Optimization cache is in-run.** `InMemoryCache` (bounded FIFO + TTL); a durable
  (Redis) cache backend is a later port. Cross-run/agent caching is not yet shared.
- **Context "compression" is illustrative** — deterministic slicing, not semantic
  summarization.
- **Semantic dedup is shingle/Jaccard** (threshold 0.85), not embedding-based.
  Cross-lingual/paraphrase detection is limited.
- **Planning/routing are advisory.** They annotate events; they do not rewrite or
  block execution. No learned/adaptive tier rerouting or cost-latency auto-tune loop.
- **Single-tenant by default.** `tenant` is tracked on policies/memories but there is
  **no enforcement, RBAC, SSO, billing, or retention** of run data yet.

**Quality/operational limits:**

- No control-plane UI; all surfaces are SDK + CLI + events.
- Demo is a synthetic cited-market-research workflow with mock tools/models; it
  validates wiring, not production model quality.
- Event schema is versioned (`0.2.0`) but not yet stabilized for external consumers;
  additive changes only so far.
- No load/scale testing; in-memory stores suit demos and tests, not high-throughput
  production traffic.

## What to validate with prospective users before the next slice

1. Does the **advisory planning + context + routing** output match how teams actually
   want to tier and scope work? (Event annotations, not actions.)
2. Is the **semantic-dedup + caching** heuristic (shingle/Jaccard, idempotency
   flags) trustworthy enough to flip to `active`? What false-positive rate is
   acceptable before serving from cache?
3. Does the **memory governance manifest** (included/excluded/stale/drift + reasons)
   give operators enough confidence to rely on retrieved context? What governance
   signals are missing?
4. Confirm the **opt-in safety model** (off by default, explicit activation) fits
   their rollout/compliance needs.

## Deferred to next slices (pending user validation)

- Cross-run (Redis) cache backend + learned/adaptive tier rerouting (remaining Phase 4
  workstream).
- Phase 5 enterprise half: multi-tenancy enforcement, RBAC, SSO/SAML/OIDC, billing/
  quotas, retention/redaction of run data.
- Control-plane UI, semantic context compression, embedding-based memory/similarity.

## Upgrade notes

- Event schema `0.2.0` is additive over `0.1.0`; existing `0.1.0` consumers keep
  working. New event types: `optimization.recommended`, `optimization.applied`,
  `memory.retrieved`, `memory.expired`.
- No breaking SDK changes vs `0.1.0`.
